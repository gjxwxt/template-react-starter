# Release Plan

## 目标定位

`template-react` 的 `v1` 目标不是 CLI，也不是运行时全动态平台，而是一个可收口、可发布、可试点接入的企业后台模板母版。

当前阶段的目标口径是：

- 先做到 `internal RC`，可供 1 到 2 个业务项目试点
- 再根据试点结果进入 `stable`

## 当前不做

以下内容不属于本轮发布目标，除非用户明确改 scope：

- CLI 生成器
- 后端返回未知 URL 后运行时自动注册页面
- 组织级低代码 / 配置化平台
- 全局主题体系的大规模重构

## 发布级别

### Internal RC

满足以下条件后，才允许进入 `RC`：

1. 代码层的 `lint / typecheck / build / smoke` 全通过
2. 模板主路径可自动化验证
3. 关键页面操作已经完成按钮级权限收口
4. 已有真实后端接入样例和文档
5. 发布清单、已知边界、tag 规则已经写清楚

### Stable

满足以下条件后，才允许从 `RC` 升到 `stable`：

1. 至少 1 到 2 个业务项目完成试点
2. 试点阻断问题已经收口
3. 试点反馈没有要求回滚模板壳层、菜单、权限主设计

## Agent 执行规则

其他 agent 必须严格遵守以下规则：

1. 开工前先读 `AGENTS.md`、`docs/README.md`、本文件、`feature_list.json`、`progress.md`、`session-handoff.md`
2. 开工前运行 `./init.sh`
3. 一次只做一个 release feature，不要并行推进多个阶段
4. 默认按本文件的 feature 顺序推进，不要跳阶段
5. 每个 feature 完成后，必须同步更新：
   - `feature_list.json`
   - `progress.md`
   - `session-handoff.md`
   - 受影响的 `docs/*.md`
6. 没跑完该 feature 规定的验证命令，不能标记为 `done`
7. 如果遇到阻塞，先在 `progress.md` 和 `session-handoff.md` 里明确写阻塞点，不要自行扩 scope 绕过去

## Feature 顺序

发布阶段严格按下面顺序执行。

### 1. `release-quality-gate`

**目标**

补齐模板级自动化质量门禁，让模板发布前不再只靠人工点击验证。

**范围**

- 为模板建立 `smoke / E2E` 验证链路
- 覆盖登录、壳层、菜单展开、页签切换、异常页、系统管理主路径
- 把 smoke 命令接入 `init.sh` 或定义等价门禁入口

**建议产出**

- 模板级 smoke 脚本或 Playwright 用例
- `package.json` 新验证脚本
- 对应运行说明文档

**验收标准**

- 存在统一的模板级 smoke 命令
- `./init.sh` 可以覆盖到该命令，或文档中明确为什么暂不并入
- 自动化至少覆盖这些链路：
  - 登录
  - 侧栏展开 / 收起
  - 多页签切换
  - 任务列表
  - 用户管理
  - 角色管理
  - 部门管理
  - 字典管理

**完成后才能进入下一阶段**

- `lint / typecheck / build / smoke` 全通过

### 2. `release-permission-closure`

**目标**

把当前权限收口缺口补齐，但保持为**可选示范能力**。修补现有页面的漏绑、错绑，同步不同角色的 mock 链路，并明确公私操作边界。

**范围**

- 修补系统管理页里“编辑”误绑到 `canCreate` 的问题
- 给任务列表、资产详情补按钮级权限接入点
- 同步 mock 菜单里 admin / operator / viewer 的动作权限码，跑通多角色权限验证
- 把“哪些动作受控、哪些动作默认公开、没有按钮权限时怎么接”写回 `ROUTING_AND_PERMISSIONS.md`

**建议产出**

- 页面级权限收口代码
- 一份权限映射说明

**验收标准**

- 模板内所有关键业务操作按钮都满足以下两类之一：
  - 已接入 `hasPermissionCode`
  - 在文档中明确声明为当前阶段不收口，且理由成立
- 受限角色下页面不会出现“看得到但点了必失败”的明显示范缺口

### 3. `release-integration-ready`

**目标**

让业务项目可以从 mock 平滑切到真实后端，不需要自己反推模板契约。

**范围**

- 登录态与接口对接
- 壳层上下文与真实信息替换
- 服务端菜单树 `type=1/2/3` 的真实契约映射
- `allowedRouteIds / navigationGroups / permissionCodes` 的消费逻辑
- 明确“不做按钮权限时怎么接”、“只做页面权限时怎么接”
- 常见接口包裹和错误码适配

**建议产出**

- 真实后端接入文档
- 最小后端响应示例
- mock -> real 切换清单

**验收标准**

- 文档中能明确回答：
  - 登录接口需要返回什么
  - 壳层上下文需要返回什么
  - 菜单树 `type=1/2/3` 怎么映射
  - 业务方第一周该改哪些文件
- 至少有一个“最小真实接入样例”可以作为业务方照抄起点

### 4. `release-publishing-ready`

**目标**

完成 `RC` 级别的发布准备，让模板进入“可以正式给业务试点使用”的状态。

**范围**

- 启动/验证/接入/已知边界文档收口
- 版本命名与 tag 规则
- 分支、MR、流水线发版约束
- 外网开发与内网迁移的源配置策略

**建议产出**

- 发布检查清单
- 版本和 tag 规则
- 模板仓库单向同步设计与导出策略
- `RC` 和 `stable` 的区别说明

**建议规则**

- 模板 tag 使用前缀，避免和组件库本体混淆
- 推荐：
  - `template-react-v0.1.0-rc.1`
  - `template-react-v0.1.0`

**验收标准**

- 新同学只看文档，就知道：
  - 怎么启动
  - 怎么切 mock / real
  - 当前边界是什么
  - 发布前要跑什么
  - RC / stable 怎么打 tag

### 5. `pilot-trial-acceptance`

**目标**

通过 1 到 2 个业务项目试点，确认模板主设计已经稳定。

**范围**

- 试点记录
- 阻断问题清单
- 修复与回归验证
- Stable 判断结论

**建议产出**

- 试点接入记录
- 阻断问题列表
- `RC -> stable` 决策记录

**验收标准**

- 至少 1 到 2 个业务项目完成模板试点
- 没有要求回滚壳层、菜单、权限主设计的阻断反馈
- 可以明确给出“继续 RC”或“进入 stable”的结论

## 每个 Feature 的统一交付要求

每个 feature 完成后，执行 agent 必须提供：

1. 变更文件列表
2. 运行过的验证命令和结果
3. 未验证项和风险
4. 下一步建议

如果 feature 包含 UI 变更，建议补一次本地浏览器手工检查，端口默认是 `5175`。

## 发布前总检查

进入 `RC` 前，必须逐项确认：

- `release-quality-gate` 已完成
- `release-permission-closure` 已完成
- `release-integration-ready` 已完成
- `release-publishing-ready` 已完成
- `feature_list.json`、`progress.md`、`session-handoff.md` 与代码现状一致
- `README.md` 与 `docs/*.md` 没有过期说明

## 推荐推进方式

当前最合理的推进顺序是：

1. 先做质量门禁
2. 再做权限收口
3. 再做真实接入样例
4. 最后做发布准备和试点验收

原因是：

- 没有质量门禁，后面每个阶段都会重复返工
- 没有权限收口，模板的企业级示范价值不完整
- 没有真实接入样例，业务方无法真正拿去落地
- 没有发布准备和试点验收，就不应该宣称“可发布使用”
