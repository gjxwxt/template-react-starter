# Routing And Permissions

## 总原则

模板采用混合模式：

- 前端本地路由表负责页面注册
- 服务端菜单树负责可见性、分组、展示名和权限码

不要把它理解成“纯静态路由”或“全动态路由”，而是更适合企业后台模板的折中方案。

## 本地路由表

路由真源在 `src/app/router/routes.tsx`。

每个路由需要关注：

- `id`：稳定的模板路由标识
- `path`：浏览器地址
- `roles`：本地角色兜底限制
- `sidebarGroup`：本地模式下的分组
- `serverRoutePath`：后端菜单 URL 与本地路由的映射键
- `serverPermissionCode`：后端菜单权限码与本地路由的映射键

## 服务端菜单树

当前模板按下面的企业接口语义适配：

- `type=1`：目录节点
- `type=2`：页面菜单节点
- `type=3`：按钮 / 操作权限节点

适配入口在 `src/app/server-menu.ts`。

## 适配输出

后端菜单树最终会被适配成：

- `allowedRouteIds`
- `navigationGroups`
- `permissionCodes`

消费位置：

- `allowedRouteIds`：控制页面是否出现在当前账号的可访问集合里
- `navigationGroups`：控制侧边栏的分组、排序和展示文案
- `permissionCodes`：给页面按钮、批量操作、页内动作使用

当前模板已经把这套模式落到系统管理和业务页样例里，以形成完整闭环：

- 用户管理：新建、编辑、删除、角色授权
- 角色管理：新建、编辑、删除、权限配置
- 部门管理：新建、编辑、删除
- 字典管理：新建、编辑、删除、字典项维护
- 任务列表：新建、编辑、批量指派、导出
- 资产详情：编辑资产

## 按钮级权限 (可选示范能力)

本模板将按钮级 `permissionCodes` 定位为**“可选示范能力”**，而不是强制的统一策略。这意味着模板天然支持以下两种用法：

1. **粗粒度 (仅页面可见性)**：如果你的业务方或后端系统不需要细粒度权限控制，前端完全可以不维护按钮级 `permissionCode`。只要在页面注册 `allowedRouteIds`，未传 `permissionCodes` 数组时模板会自动放行所有按钮。
2. **细粒度 (按钮与操作收口)**：如果需要细粒度管控，后端可通过 `type=3` 节点下发动作权限码。通过 `hasPermissionCode(session, 'Domain:action')` 即可将非法操作按钮在视图层隐藏。

### 明确不收口的公开操作

为了避免过度设计，以下几类操作属于“默认公开动作”，不需要且不应该绑定专属的 permissionCode：

- **搜索与重置**：表格顶部的 `SearchForm` 查询。由于进入页面本身已经过了页面级权限控制，正常的列表检索无需二次授权。
- **导航与回退**：如资产详情页的 `返回`，面包屑导航等。
- **视图切换与过滤**：如 Dashboard 页面的时间范围切换（按小时/天/月）。这些仅仅是对已授权数据的视图展现形式进行改变。
- **纯详情查看**：如不包含表单提交的只读抽屉/弹窗展示。

## 为什么不做全动态路由

模板默认不做“后端返回一个新地址，前端运行时自动挂一个未知页面”，原因是：

1. 模板需要稳定的页面注册和异常页策略
2. 业务项目通常还是有明确的页面代码边界
3. AI 和团队成员更容易定位问题和修改入口
4. 这样更接近现有 Vue 模板和大量企业后台项目的落地方式

## 接入真实后端时怎么做

1. 先在 `src/app/router/routes.tsx` 注册本地页面
2. 给新页面补 `serverRoutePath` 或 `serverPermissionCode`
3. 让后端菜单树返回 `type=1/2/3` 结构
4. 在壳层上下文接口里落成 `allowedRouteIds + navigationGroups + permissionCodes`
5. 页面操作区逐步消费 `hasPermissionCode`

## 当前已知边界

- 复杂外链、iframe、跨系统 tab 打开策略还没有抽成模板规范
- 由于 `hasPermissionCode` 依赖 `AppSession`，所以目前仅适用于受用户上下文约束的内部路由，对匿名公开页无效
